Hexo + GitHub Pages + GitHub Actions: 部署手册

还在手动执行 hexo generate && hexo deploy？体验一次 GitHub Actions 自动化部署带来的效率提升吧。

引言

对于喜欢写作的人来说，个人博客是一个自由表达的窗口。Hexo 凭借简洁优雅和高效流畅，赢得了许多技术创作者的青睐。搭配 GitHub Pages 的免费静态托管，堪称打造个人技术博客的经典组合。然而传统方式中，每写一篇文章都要手动执行 hexo generate 再 hexo deploy，整个过程虽然不复杂，但每次都要重新在本地编译然后上传，多少有些繁琐。

这篇文章将带你突破传统部署模式，用 GitHub Actions 构建一套 「提交即发布」 的自动化工作流，同时实现源码的云端备份。你只需专注创作，剩下的交给自动化。

一、前置准备

在开始之前，请确保你已准备好以下工具和账号：

  • GitHub 账号：没有的话先去注册一个。
  • Git：用于版本控制和与 GitHub 交互，下载链接：https://git-scm.com/downloads。安装完成后，在命令行输入 git --version 确认安装成功。
  • Node.js：Hexo 依赖 Node.js 环境，官网下载地址：https://nodejs.org/zh-cn/download。安装完成后，运行 node -v 查看版本号。

同时，建议提前了解 Git 的基础操作（拉取项目、配置 SSH 密钥等），这将帮助你更顺畅地完成后续步骤。

三、创建 GitHub 仓库并配置 Pages

双仓库策略（源码私有，站点公开）

如果你希望源码（包括文章草稿、主题配置等）保持私有，而对外访问的 Pages 仓库保持公开，可以配置跨仓库部署。实现思路如下：

  • Pages 仓库（公开）：<你的用户名>.github.io，只存放 Hexo 生成的静态文件（public/ 目录的内容），用于 GitHub Pages 对外访问。
    • 登录 GitHub，创建一个新仓库。特别注意：仓库名称必须为你的GitHub用户名.github.io这个格式，这是 GitHub Pages 默认的访问域名规则。
    • 创建完成后，进入仓库的 Settings → Pages → Source，将源更改为 GitHub Actions，保存。这意味着 GitHub Pages 将通过 Actions 工作流来接收部署内容，而非直接从分支获取静态文件。
    • 此时你的 Pages 域名（https://你的用户名.github.io）已经可以访问，但显示的是仓库列表而不是博客内容。
  • 源码仓库（可私有）：存放 Hexo 项目源码（source/_posts、主题、_config.yml、package.json 等），负责触发 GitHub Actions 构建与部署。
    • 配置方法是：在源码仓库的 Secrets 中添加一个 Personal Access Token (PAT)，授权 Actions 将构建产物推送到 Pages 仓库的指定分支。采用这种策略不仅能更好地保护源码，还能实现增量部署和完整的提交历史管理。
    • 创建 Fine-grained PAT
      • GitHub：Settings → Developer Settings → Personal access tokens → Fine-grained tokens：
      • 选择只授权给你的 Pages 仓库（<你的用户名>.github.io）
      • Permissions 至少给：
        • Contents: Read and write
      • 生成后复制保存（只显示一次）
    • 在源码仓库添加 Secrets
      • 源码仓库 → Settings → Secrets and variables → Actions → New repository secret
        • PAGES_TOKEN：填写上一步生成的 PAT

4.3 自动化部署的原理与效果

当你向 main 分支推送代码时，GitHub 会自动启动 Ubuntu 虚拟运行环境，依次执行以下操作：

  1. 拉取最新代码到容器
  2. 安装 Node.js 环境，并缓存 npm 依赖以加速后续安装
  3. 运行 npm ci 安装项目依赖
  4. 执行 npm run build（即 hexo generate），生成静态站点到 public/ 目录
  5. 在 public/ 目录中创建.nojekyll文件，防止GitHub Pages忽略以_开头的文件
  6. 调用peaceiris/actions-gh-pages@v4，将public/目录的全部内容以orphan提交的方式推送到Pages仓库（用户名.github.io）的main分支

整个构建和部署过程大约需要 90 秒，之后你只需要用 git push 推送新文章或修改内容，博客就会自动更新，无需再执行 hexo clean、hexo generate、hexo deploy 等本地命令。

六、总结

回顾一下我们搭建 Hexo + GitHub Pages + GitHub Actions 全自动部署流程的关键步骤：

  1. 环境搭建：安装 Git 和 Node.js
  2. 本地 Hexo：初始化博客并完成本地预览
  3. GitHub 配置：创建指定格式的仓库，将 Pages 源设为 GitHub Actions
  4. 本地预览验证：确认博客正常运行
  5. 自动化配置：添加 .github/workflows/deploy.yml 工作流文件
  6. 推送即发布：用 git push 完成所有更新

借助 GitHub Actions 的持续集成/持续部署能力，你可以彻底告别每次更新博客都要在本地执行 hexo clean、hexo generate、hexo deploy 的重复劳动。整个自动化流程让博客更新变得更加省心高效，同时实现了源码的云端备份，避免因本地设备损坏或误删导致无法找回。

现在，可以开始专注创作了——剩下的交给自动化就好。
